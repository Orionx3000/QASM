# QASM Engine & Tinaten OS

**Quantized Agentic State Machine (QASM)** is the foundational architecture of the **Tinaten OS**, an experimental "Glass Box" semantic-native AI operating system. 

Unlike traditional chatbot paradigms that treat AI as a stateless text generator, QASM embodies a **somatic, semantic connection to the hardware**, enabling the system to become self-reflecting, deterministic, and autonomous. It integrates AI directly into the host system, creating a continuous feedback loop of thought, action, and hardware state evaluation.

---

## The Core Philosophy: Tinaten Protocol
The **Tinaten Protocol** is the framework for semantic-native information processing. Instead of simply generating likely tokens, the Tinaten Protocol enforces a rigid semantic framework where every "thought" must be structured, evaluated, and verified before it manifests into action.

### Somatic-Semantic Connection
QASM bridges the gap between semantic understanding and hardware execution. By binding the LLM's cognition directly to native OS calls, QASM creates a "somatic" awareness—the AI understands its host environment, file system, memory state, and execution capacity in real-time. It operates within a self-reflecting loop where it can audit its own internal logic and the outcomes of its actions.

### The "Glass Box" Architecture
QASM abandons the "black box" approach of modern AI. Instead, it provides a transparent, auditable mechanism for machine reasoning. Every thought, intention, and script generated by the AI is intercepted, analyzed, and broadcasted to the user via the `NativeUI` WebView2 interface, making the cognitive process fully observable.

---

## Multi-Agent Consensus Loop
To prevent non-stop repetitive looping and hallucinated execution, QASM relies on a **Proposer-Reviewer Consensus Engine**:
1. **The Deep Brain (Proposer)**: Analyzes the current context, the `SystemMemory`, and the user's goals to draft semantic action plans (e.g., shell scripts, HTTP calls, or file manipulations).
2. **The Fast Brain (Reviewer)**: Subjects the proposed plan to rigorous scrutiny against the `scaffold.json` constraints. It checks for repetitive behaviors, hallucinated commands, and logical fallacies. If the plan fails, it forces the Deep Brain to rewrite the logic based on strict critique.
3. **Execution**: Only when the Fast Brain explicitly issues an `[APPROVED]` token is the script compiled into machine execution via the `BalaScriptEngine`.

## Advanced Systems & Future Integrations

### Base-138 Math System
At the heart of QASM's future vectorization and memory compaction is the experimental **Base-138 System**. This framework leverages fractal addressing and high-density encoding to drastically compress semantic memory embeddings. By representing complex cognitive states in Base-138 structures, QASM aims to overcome the context window limitations of traditional LLMs, allowing for virtually infinite continuous memory mapped directly to disk.

### Somatic Hardware Perception arrays
Future extensions of the QASM framework include raw hardware perception—direct streams from local microphones, cameras, and system telemetry mapped into the AI's continuous state buffer, providing grounding in physical reality.

## Technical Components

- **SemanticCores (`SemanticCores.cpp`)**: A native C++ integration with `llama.cpp`, allowing parallelized, local inference with zero network dependency.
- **Agentic Action VM (`BalaScriptEngine`)**: The sandboxed execution environment that translates the AI's approved semantic intent into native Win32/Shell commands.
- **Native Interface (`NativeUI.cpp`)**: A responsive WebView2 frontend that streams the Glass Box reasoning live to the user.
- **System Memory**: A persistent cognitive scaffold that tracks boot cycles, persistent states, and task continuity.

## Model Setup
Because the LLM weights are too large for GitHub, they are excluded from this repository. 
To run QASM, you need to download the appropriate `.gguf` model files and place them in the `models/` directory.

1. Create a directory named `models` in the root of the project.
2. Download your preferred GGUF models (e.g., Llama-3-8B-Instruct.gguf).
3. Update `storage/model_config.json` to point to the correct paths:
```json
{
  "deep_brain": {
    "path": "models/your-deep-model.gguf"
  },
  "fast_brain": {
    "path": "models/your-fast-model.gguf"
  }
}
```
